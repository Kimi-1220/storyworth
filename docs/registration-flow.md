# 登録の導線設計（何もない状態 → 執筆開始まで）

## このドキュメントの狙い

「サインアップから最初の1問を書き始めるまで」をできるだけ**少ないステップ・明確な主語**で
設計する。`concept.md` は贈り物モデル（子・孫 → 親・祖父母）前提だが、実際の入口は
「**だれの物語を本にするか**」で大きく二手に分かれる。ここを最初の1画面で分岐させる。

関連: [concept.md](./concept.md) / [writing-studio.md](./writing-studio.md) / [line-experience.md](./line-experience.md) / [questions-by-audience.md](./questions-by-audience.md)

---

## 核心の発見：分岐の正体は「誰が答えるか」

| | 口座の持ち主 | 本の主役 | **質問に答える人** | 状態 |
| --- | --- | --- | --- | --- |
| **A. 自分の自伝** | 本人 | 本人 | 本人 | MVP対象 |
| **B. 家族への贈り物** | 子・孫（贈り主） | 親・祖父母 | **親・祖父母** | MVP対象 |
| C. 聞き書き・追悼 | 子・孫 | 親・祖父母（故人可） | **子・孫** | 将来（質問の主語が反転するため別テンプレが必要） |

> **A と B は「答える人＝主役本人」なので、質問文がそっくり同じ（主役への二人称）になる。**
> 「あなたが初めて子どもを抱いたとき、何を思いましたか？」は自分でも親でもそのまま使える。
> 分岐しているのは**質問文ではなく、招待・通知のセットアップだけ**。
> （C だけは「お母さんはどんな人でしたか？」と主語が反転するので難易度が一段違う → 後回し）

### 設計の前提（決定事項）

- **対応範囲**: A（自分の自伝）＋ B（家族への贈り物）。C は将来拡張。
- **課金位置**: **先に購入**（ギフト前提）。執筆前に決済を挟む。
  → トレードオフ: 「最短で書き始める」原則と相反するので、**決済前に価値を体感させる仕掛け**
    （サンプルの問い・本のプレビュー見本・できあがりイメージ）を必ずLP/オンボーディングに入れる。
- **アカウントは決済の前に明示的に作る**: 購入者のメール＋フルネームを最初に取得してアカウントを作る。
  Stripe にメールを集めさせる方式だと「どのアドレスでアカウントが出来たか分からない」事故が起きるため、
  **自前で先にメールを取り、それをアカウントのアンカーにする**（→ 前回の「ログイン前購入の紐付け」問題の解）。
- **通知チャネル**: LINE 中心。ただし A（自分の自伝）では**メールも選べる**。
  通知層はベンダーロックを避け差し替え可能に保つ（→ writing-studio.md / line-experience.md）。

## 主語を明確にする原則（このプロダクトの肝）

質問文・コピーは「答える人＝主役」への二人称で統一する。導線の各画面で
「**いま操作しているのは誰か / 本の主役は誰か / 次に答えるのは誰か**」を常に明示する。

- 贈り物（B）では途中で**操作する人が贈り主から語り手へ移る**。この受け渡しが導線の山場。
- 画面コピーは「あなたの本」「あなたの自伝の取材」「著者」で揃える（→ writing-studio.md「著者気分」）。

## 購入の導線（確定順序）

購入は **プラン → 用途フォーク → アカウント →［ギフト時のみ相手情報］→ 部数 → 決済 → オンボ** の順。
各ステップで「いま誰の何を入力しているか」を明示し、相手の連絡先のような“面倒で後回しにできるもの”は
チェックアウトから外してオンボへ送る。

```
[1] プランを選ぶ（最初の意思決定＝価格）
    ・ベーシック : モノクロ写真・テキスト回答のみ（音声なし）
    ・スタンダード: カラー写真・音声回答あり
    ・プレミアム : 毎週プロのインタビュアーが電話で取材
        │ ※プラン名は仮。差分（モノクロ/カラー・音声有無・電話取材）が要点
        ▼
[2] 用途フォーク：自分用 / ギフト用
    ・自分自身の物語（自伝）
    ・家族の物語（→ 続柄：母/父/祖母/祖父/その他 → 対象者別の質問セット確定）
        │
        ▼
[3] 購入者のメール ＋ フルネーム → アカウント作成（この時点は未決済）
    ・自分用でもギフト用でも必ずここで取る。アカウントのアンカー＝このメール
    ・Stripe 任せにしない（どのアドレスで作ったか不明になる事故を防ぐ）
    ・誤入力対策に確認表示／2回入力。検証自体は決済後の初回マジックリンクで回収
        │
        ▼
[4]【ギフト時のみ】相手の情報（※連絡先はまだ聞かない）
    ・送る相手のフルネーム
    ・送信手段の選択（LINE / メール / 印刷して手渡し など）
    ・送信文言（デフォルト文が prefill 済み・編集可）
    └ 狙い: 「こうやって相手に届くんだ」を理解させる“予告”。
       実際のアドレス/LINEは面倒なので後（オンボ/招待）で取得する
        │
        ▼
[5] 発行部数を選ぶ（価格の数量ライン）
        │
        ▼
[6] Stripe でカード入力 → 購入（アカウントを paid に）
        │
        ▼
[7] オンボーディングへ
```

### 自分用（A）と ギフト用（B）の分岐点

- **[2] と [4] だけが分岐**。それ以外（プラン・アカウント・部数・決済）は共通の1本道。
- **自分用（A）**: [4] をスキップ。決済後のオンボ＝そのまま本人がスタジオに着地し、1問目を書く。
  週次の問いの受け取りチャネル（LINE / メール）はオンボで選ぶ。
- **ギフト用（B）**: 決済後のオンボで初めて**相手の実際の連絡先**（[4]で選んだ手段に応じて LINE 連携コード/QR、
  またはメール、または印刷用案内）を用意し、語り手へ届ける。ここから**主語が語り手へ移り**、
  語り手はマジックリンクでスタジオに着地 →「あなたの自伝の取材です」のオンボ → 1問目。

### プレミアム（電話取材）の連絡先

プロが電話するには語り手（自分用なら本人）の**電話番号と曜日・時間**が要るが、これも[4]では聞かない。
方針通り**招待が繋がった後＝オンボで取得**し、チェックアウトを軽く保つ。

## ペルソナ別オンボーディングの要点（決済後）

- **自分の自伝（A）**: 動機あり・操作できる。摩擦ゼロで即執筆。「書く/話す」「あとで直せる」を最初の1問で体感
  （→ writing-studio.md）。週次チャネル（LINE/メール）をここで選ぶ。
- **贈り主（B）**: 機械が苦手な親でも続くか不安。[4]で選んだ手段で語り手へ届ける段取りをオンボで完了させ、
  「親が受け取る画面」「できあがる本の見本」で安心させる。1問目を選ぶ or おまかせ。
- **語り手（B・60〜90代）**: 導線の最難関。マジックリンクで着地直後にログイン済み・アカウント作成画面を見せない。
  大きな文字・「あなたの自伝の取材」で著者に祭り上げる（→ writing-studio.md「着地の設計」）。

## データモデルへの示唆

現状の `prisma/schema.prisma` は `Storyteller`（＝答える人）単体で、贈り主や「本人/家族」の区別を持たない。
A/B を綺麗に扱うには次が要る（roadmap フェーズ3「認証」相当）。

- **アカウント（口座の持ち主）と語り手（主役）を分離**する。アンカーは `Account.email`（[3]で取得）。
  - Path A: アカウント＝語り手（同一人物）。`kind = "self"`。
  - Path B: アカウント（贈り主）が語り手を1人以上所有。`kind = "gift"`。
  - アカウントは**決済前に作られる**ので `status`（unpaid / paid）を持つ。unpaid はカゴ落ち再訪・再送に使える。
- `Storyteller.relationship`（self / 母 / 父 / 祖母 / 祖父 / その他）で**質問セットと主語の出し分け**。
- **ギフトの“予告”情報を語り手に持たせる**: `inviteMethod`（line / email / print）と `inviteMessage`（prefill編集後）。
  [4]で選ぶが実際の連絡先はまだ無いので、`status = pending-invite` で保持し、オンボで連絡先が埋まる。
- `Storyteller.notifyChannel`（line / email）。A はメール可、B は LINE 既定。プレミアムは電話番号＋曜日時間（オンボで取得）。
- 購入レコード（`plan` = basic/standard/premium、`copies` = 発行部数、決済状態、開始日、Stripe セッションID）。

> いずれもこの導線が固まってから着手する範囲。ここでは「導線がこの分割を要求する」ことだけ記録する。

## 未解決・要検討（トレードオフ）

- **先払い × 最短執筆の両立**: 決済を挟む分、決済前の「価値の見せ方」（サンプルの問い・本のプレビュー）
  の作り込みが完成率を左右する。ここが弱いと離脱する。
- **メールの誤入力検証**: アカウントを決済前に作るので、誤入力アドレスはマジックリンクが届かず詰む。
  funnel は止めず、決済後の初回マジックリンク＋「届かない？再送/修正」で回収する（[3]に確認表示/2回入力）。
- **未決済アカウントの掃除**: Stripe 離脱で unpaid アカウントが溜まる。カゴ落ちメールに使いつつ、期限切れ整理の方針が要る。
- **C（聞き書き・追悼）の扱い**: 需要は確実にある（故人・要介護で本人が書けない）が、質問の主語が反転する。
  別テンプレ＆別オンボーディングが必要なので将来拡張として切り出す。
- **語り手が複数人**（両親それぞれに贈る等）: アカウントが複数 Storyteller を持つ前提で破綻しないか。
- **A→B / B→A の乗り換え**: 自分用に始めた人が親にも贈る、などの動線を後から足せる構造にしておく。
